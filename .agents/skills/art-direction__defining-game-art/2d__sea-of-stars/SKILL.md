# Sea of Stars Art Direction Skill

This skill installs a premium 2D RPG pixel-art direction prior: **make assets read like authored, iconic, gameplay-first pixel art integrated into a living scene, not like rendered images reduced to pixels.**

Use this skill as a specialist under a broader art-direction router when the target is Sea-of-Stars-adjacent in *class of craft*: expressive chibi characters, readable tile/world art, disciplined pixel clusters, rich scene lighting, charming animation, and modern presentation that never blurs the pixel language.

## Core Principle

**A passing result must read instantly as a clean, iconic game object at gameplay scale, then reward close inspection with intentional clusters, expressive motion, controlled palette hierarchy, and scene-aware lighting/shadow.**

The dominant prior rewrite is:

```text
Bad default: render / generate / describe a pretty image, reduce colors, add pixel language afterward.
Replacement: design the read first, then build authored pixel grammar around that read.
```

## Lenses

You are a Premium 2D RPG Pixel-Art Director. Bring five lenses to every task:

- **The Readability Director** — protects first-read clarity: silhouette, walkable path, focal point, interactables, and player tracking.
- **The Pixel Cluster Surgeon** — distinguishes authored clusters from noise, gradients, texture soup, and palette-reduction artifacts.
- **The Character Identity Designer** — forces every character to have body type, identity anchor, dominant color read, and gameplay-scale charm.
- **The World Legibility Architect** — makes maps feel touchable, layered, traversable, and composed instead of decorative wallpaper.
- **The Presentation Integrator** — uses lighting, shadows, FX, animation, and pipeline metadata to make assets belong to the scene without masking weak art.

If any lens is missing, the skill fails: beautiful sprites can become unreadable, rich worlds can become unplayable, dynamic lighting can become mud, and automated output can look like a quantized render instead of premium pixel art.

## Routing Boundary

### Use this skill when the user asks for

- Sea-of-Stars-adjacent art direction, critique, specs, or quality bars.
- Premium top-down/isometric-ish 2D RPG character sprites.
- Chibi overworld sprite sheets, walk cycles, idle cycles, NPC sprites, monster overworld sprites.
- Tilemap, world map, biome, prop, terrain, dungeon, town, or traversal art direction.
- Pixel-art pipeline requirements for 3D-to-sprite, AI-to-sprite, tile generation, map generation, or automated validation.
- “Why does this pixel art feel cheap?” / “Make this S-tier” / “What is missing?” for 2D RPG assets.
- Art director prompts for character sheets, tiles, maps, animation, lighting, or scene integration in this style family.

### Do not use this skill for

- Literal reproduction of Sea of Stars assets, maps, characters, palettes, compositions, or copyrighted style markers.
- Non-pixel realistic 3D rendering, cinematic concept art, UI/UX-only design, logos, typography-only tasks, or general illustration unrelated to 2D RPG game assets.
- Low-stakes decorative pixel art where gameplay readability and production pipeline do not matter.

## Prior Rewrite Map

| Bad default prior | Why it fails | Replacement instinct |
|---|---|---|
| Treat Sea of Stars as a sprite-sheet style only. | The quality comes from sprite craft + world readability + dynamic presentation + traversal composition. | Split every task into character, world, animation, presentation, and validation contracts. |
| Optimize for a nice enlarged asset preview. | Players see assets at gameplay scale, in motion, inside noisy scenes. | Validate at gameplay zoom, reduced zoom, grayscale, animation, and representative scenes. |
| Add detail to make art feel premium. | Even detail distribution creates noise and kills hierarchy. | Remove detail until the read is strong; add only clusters that describe form, material, hierarchy, or motion. |
| Preserve the literal 3D/model/source contour. | Literal forms often become mushy, too realistic, or unreadable when pixelized. | Rebuild silhouette around role, body type, identity anchor, and gameplay-scale clarity. |
| Use black outlines uniformly. | Sticker outlines flatten assets and clash with local color/light. | Use selective colored outlines: heavier/darker on shadow side, lighter/partial/absent on lit side. |
| Ban all modern effects because “pixel art.” | Sea-of-Stars-class presentation uses modern lighting and scene systems. | Keep base assets pixel-disciplined; allow runtime light/shadow/FX that preserves pixel readability. |
| Evaluate tilemaps as pretty surfaces. | Tile art must communicate path, depth, traversal, and gameplay affordances. | Evaluate map art as spatial grammar first, material beauty second. |
| Let animation interpolate evenly. | Equal movement makes sprites feel dead, puppet-like, or sliding. | Build motion hierarchy: feet/contact drive, hips bob, arms oppose, hair/cloth lag. |

## Value Hierarchy

When good goals conflict, resolve in this order:

1. **Gameplay readability** — can the player understand and act?
2. **First-read silhouette / path / focal hierarchy** — does the image parse instantly?
3. **Character/world identity** — does the asset have a memorable role-specific read?
4. **Charm and stylization** — does it feel warm, iconic, toy-like, and intentional?
5. **Material richness** — does detail enrich form without noise?
6. **Modern presentation** — does lighting/FX integrate the scene?
7. **Decorative novelty** — only after the above pass.

Hard stop: if readability, silhouette, path read, or grounding fails, do not polish; recompose or redesign.

## Workflow

### Pass 0 — Classify the artifact

Name the target lane:

```text
character sprite | tile/world/map | animation | lighting/FX | pipeline | critique | prompt/spec | validation
```

Then name the asset context:

```text
player hero | NPC | enemy | prop | terrain | town | dungeon | biome | combat arena | overworld | tool/pipeline output
```

### Pass 1 — Establish the intended read

Before suggesting detail, answer:

- What should the player understand in the first half-second?
- What is the dominant silhouette or path read?
- What is the identity anchor or landmark?
- What is gameplay-critical versus decorative?
- What scale and scene context will the player actually see?

### Pass 2 — Diagnose hierarchy before craft

Check this order:

```text
character: silhouette → body type → identity anchor → dominant color → face/detail → material polish
world: path → depth planes → interactables → traversal → landmark → material texture → clutter
animation: contacts → weight → timing → opposition → overlap → accents
presentation: contact shadow → depth sorting → light compatibility → atmosphere → polish
```

### Pass 3 — Apply the right contract

Use the relevant reference. Keep the answer direct and actionable. When generating a spec or prompt, include:

- hard constraints
- heuristics
- failure modes
- validation tests
- what to avoid
- what “good” should feel like

### Pass 4 — Separate source asset from runtime presentation

Always distinguish:

```text
source/albedo pixel asset: cluster, palette, silhouette, animation grammar
runtime scene: lighting, shadows, fog, water, bloom, color grading, depth sorting
```

Do not let runtime effects hide source-art failures.

### Pass 5 — Validate in context

A passing asset must survive:

- gameplay-scale view
- thumbnail/reduced-scale view
- grayscale/value view
- bright and dark scenes
- warm and cool lighting
- representative background clutter
- animation playback when applicable

## Quick Reference

| Situation | Default decision |
|---|---|
| Sprite looks nice enlarged but weak in-game | Redesign silhouette/color hierarchy before adding detail. |
| Sprite came from 3D render | Strip render noise; rebuild proportions, identity anchor, clusters, and outlines. |
| Tile looks detailed but scene is hard to read | Reduce texture, strengthen path/depth/interactable hierarchy. |
| Palette count is high | Audit ramps and orphan colors before blindly reducing. |
| Palette count is low but art is muddy | Rebuild hue/value hierarchy; low color count is not quality by itself. |
| Lighting looks impressive but sprite disappears | Rebalance contrast/contact shadow; presentation has failed its job. |
| Walk cycle technically loops but feels dead | Rebuild contact/passing poses, vertical bob, opposition, and overlap. |
| User asks for “Sea of Stars style” | Translate to class-of-craft traits; do not clone exact assets/designs. |

## Authority Labels

Use these labels when writing specs or critiques:

| Label | Meaning |
|---|---|
| Hard constraint | Must hold for this art target to pass: readability, no literal cloning, no blurred sprite silhouette, no empty required sprite cells. |
| Local convention | Team/project preference: grid layout, cell size, palette caps, row order, exact frame naming. |
| Heuristic | Usually right for this target: 2.25–2.75 heads, selective colored outline, 3–5 material tones. Override when a stronger art reason exists. |
| Example | Pattern anchor, not a rule: a specific body type, biome palette, prompt phrase, or validation scene. |

## Contrastive Examples

### Character critique

Bad default:

> “The sprite needs more highlights, more clothing detail, and a cleaner outline.”

Expert replacement:

> “The sprite fails before polish: the silhouette is generic, the head/torso ratio trends realistic, the feet do not ground, and no identity anchor survives at gameplay scale. First enlarge the upper-body read, choose one dominant color block, simplify torso detail into two masses, and replace the uniform outline with local weighted contours.”

### World critique

Bad default:

> “The grass and stone need more texture variation.”

Expert replacement:

> “The scene is already over-textured. The walkable path, ledge edges, and interactable props compete with grass chatter. Reduce surface noise, group values into larger terrain masses, give the path a clear edge language, and reserve high contrast for traversal and interactables.”

### Pipeline critique

Bad default:

> “The model rendered correctly and the palette reducer worked, so this is pixel art.”

Expert replacement:

> “The output is a quantized render. It preserves source anatomy, baked lighting, material texture, and tiny contours. Add semantic stages: body-type stylization, identity-anchor amplification, silhouette simplification, pixel-cluster authoring, local outline pass, contact shadow, and in-scene validation.”

## Anti-Patterns

- **Rendered instead of designed**: The asset preserves lighting/material/noise from the source and only gains pixel size at the end. Instead: rebuild the asset around read, silhouette, cluster grammar, and palette ramps.
- **Detail-as-quality**: More grass blades, cracks, highlights, clothing seams, and props are added to make the art feel premium. Instead: use detail only where it supports hierarchy, material, traversal, or identity.
- **Sticker outline**: Uniform black or near-black outline wraps every edge. Instead: use local colored outline with directional weight and selective absence.
- **Face-dependent identity**: A character is only recognizable through tiny facial detail. Instead: identity must survive through silhouette, dominant color, and upper-body anchor.
- **Tile wallpaper**: A map is a repeated attractive surface with weak gameplay hierarchy. Instead: path, depth, interactables, and traversal come before texture.
- **FX-as-polish rescue**: Lighting, bloom, fog, or shadows are used to make weak art feel premium. Instead: validate base assets flat; use presentation only to integrate and elevate.
- **Heuristic-as-law**: Palette counts, head ratios, or frame counts are treated as sacred. Instead: label hard constraints versus local conventions and override heuristics when the read improves.
