# Team Cognition and Transactive Memory

Use this when designing raids, dungeons, PvP squads, logistics teams, guild operations, AI-human parties, leadership tools, or wipe/after-action systems.

## Core prior rewrite

Bad default: group design means roles, chat, pings, and ready checks.

Replacement: strong teams need shared mental models and transactive memory. Players need to know the plan, the task model, who knows what, who owns which calls, and how the team updates its model after failure.

## Coordination surfaces

| Need | Surface |
|---|---|
| shared task model | plan preview, route map, phase objective, mechanic priority |
| transactive memory | who knows route, who has cleanse, who calls interrupts, who has done phase 3 |
| role ownership | declared caller, backup caller, marker owner, logistics lead |
| readiness | readiness by mechanic, resources, knowledge, and time, not just “ready?” |
| repair | post-failure model update: what assumption failed? |
| substitution | backup roles and known competencies |
| learning | replay, examples, mechanic-specific assignments |

## Design questions

1. Does the group know the same plan?
2. Does the group know who knows what?
3. Can a leader delegate without voice?
4. Can a new player safely own a small responsibility?
5. Does a wipe recap repair the shared model or prosecute individuals?
6. Does the AI assistant improve the team’s model or merely issue commands?

## AI-native use

AI can act as:
- planner summarizer;
- mechanic explainer;
- transactive-memory surface;
- logistics coordinator;
- after-action scribe;
- mentor matching assistant.

But AI must not become an opaque commander that players obey without understanding.
